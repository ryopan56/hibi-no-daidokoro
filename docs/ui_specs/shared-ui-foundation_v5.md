# shared-ui-foundation v5

## 概要

- Task12「共通 UI 基盤とレイアウトシェル整備」の Pencil フロー用 mock v5
- v3 の静かさ、やわらかさ、ほっとする感じを維持しつつ、固定光だまりレイヤーの導入と Startup Copy Layer の軽量化を行った版
- 今回の修正版では、見た目の方向性は維持したまま、auth / home / ナビゲーションの導線配置を整理した
- artifact-slug: `shared-ui-foundation`
- version: `v5`

## v3 からの変更点

- `Startup Copy Layer` をカード的な見え方からさらに離し、真っ白に近い背景の中へ文字だけがふわっと現れる前提へ寄せた
- auth / signed-in に残っていた肯定メッセージ box を削るか極小化し、肯定メッセージの主役を Startup 側に集約した
- 全フレームに共通で使える固定の `光だまりレイヤー` を追加し、地味さを少し和らげながら静かな温度感を足した
- flash / error は光だまりの上に直接置かず、共通の面の中に整理して役割が読める見え方を維持した

## 今回の v5 修正点

- auth の `signup` 導線を、主 CTA 下の補助リンクとして見やすくした
- signed-in shell に、`/home` から日付単位ログ一覧へつながる受け皿を追加した
- home 上部は常設歓迎 box ではなく、一時メッセージ領域として読める見え方へ整理した
- ボトムナビの役割を `Home / Calendar / Search` に寄せ、高頻度導線だけを置く前提を明示した

## フレーム構成

1. `Startup Copy Layer`
2. `Auth Shell`
3. `Signed-in Shell`
4. `Flash Message Pattern`
5. `Form Error Pattern`

## Startup Copy Layer をどう整理したか

- v3 よりも前段レイヤーとしての軽さを優先し、独立したカードとしては扱わない
- 真っ白に近い背景に近い面の中へ、短いコピーがふわっと現れて消える前提を示す
- 起動時に登録済みテキストからランダム表示され、その後に auth または signed-in shell が出ることだけを見せる
- 秒数や詳細アニメーションは、この mock の段階では固定しない

## auth / home から何を削ったか、弱めたか

- auth の肯定メッセージ box は置かず、導入は短いテキストだけにした
- signed-in の歓迎 box も常設しない
- signed-in では `primary action` を最初に見せ、二次導線と下部ナビは受け皿に留める
- 肯定感は Startup で先に渡し、その後の shell では行動と案内を中心にする

## signup 導線をどう改善したか

- auth 右上の補助導線は残しつつ、それだけに依存しないようにした
- 主 CTA `ログインする` の直下に、少し小さめの補助ボタンとして `初めての方はこちら` を追加した
- 役割はあくまで補助導線であり、主 CTA の優先順位は変えない

## home の一覧受け皿をどう置いたか

- signed-in shell の `primary action` の下に、日付単位ログ一覧へつながる領域を置いた
- 一覧詳細そのものは作り込まず、日付行が並ぶ受け皿だけを示した
- `/home` が歓迎だけの画面ではなく、日付単位ログ一覧へつながる入口であることが読める構成にした

## 固定光だまりレイヤーの意図

- v3 の「少し地味」という印象を、装飾ではなく空気の温度差でやわらげるために導入した
- 模様や円形装飾にはせず、にじみや光のたまりのような曖昧な見え方に留める
- 背景として常に存在するが、意味を持つ主役にはしない

## 光だまりの位置と強さの考え方

- 位置は全画面共通で 2 か所までに抑えた
- 右上にやや温かい光、左下に淡いクリーム系の光を置き、UI の主導線を邪魔しない構成にした
- どちらも blur と低い不透明度で処理し、明確な円や模様に見えない強さへ落としている
- auth / signed-in / flash / error のどのフレームでも、読ませたい面より後ろに退くことを優先した

## flash / error との関係

- 肯定メッセージは Startup 側へ集約し、常設の flash と混ぜない
- `success flash` は操作結果の通知
- `info / guidance` は補助案内
- `form error` は修正ポイントの提示
- いずれも光だまりの上へ直接置かず、共通のサーフェス面で受けて読みやすさを保つ

## 下部ナビをどこまで確定したか

- v5 でも「受け皿」まで
- 高頻度導線として `Home / Calendar / Search` を載せる前提を mock 上で示した
- 通知設定 / Export / Import はボトムナビに載せず、設定ハブや右上導線側で扱う想定とした
- 最終ラベル、詳細遷移、設定ハブの完成形までは確定しない

## 設定系導線の考え方

- 高頻度ではない設定系導線はボトムナビから外す
- 候補は通知設定 / Export / Import などで、右上の設定ハブ側で扱う前提にする
- mock では詳細画面を作らず、home 側の補助説明でその分担だけを示す
- 右上導線は、通知設定 / Export / Import を含む設定ハブ想定として扱う

## 実装時の注意点

- 固定光だまりレイヤーは共通 base / shell 側で管理し、個別画面ごとに位置や強さを増やしすぎない
- CSS 実装時も blur と opacity を抑え、UI より前に出さない
- Startup Copy Layer は一時表示レイヤーとして扱い、shell 内の常設メッセージで代替しない
- auth / signed-in では主 CTA と補助導線の優先順位を崩さない
- auth の `signup` は主 CTA 下の補助ボタンとして実装し、主 CTA と競合させない
- home の一時メッセージ領域と日付一覧受け皿は別役割として分ける
- ボトムナビには高頻度導線だけを置き、設定系は右上ハブや設定画面へ逃がす
- Task12 の段階なので、個別画面の完成度を上げに行かず、共通受け皿の温度感づくりに集中する
- `login_id` / `display_name` 前提は維持する

## 承認時に見たいポイント

- 光だまりが温度感を足している一方で、主張しすぎていないか
- Startup Copy Layer が v3 より軽く、前段レイヤーとして読めるか
- auth / signed-in から肯定 box を減らしても冷たくなっていないか
- flash / error が背景の気配に埋もれず読めるか
- 下部ナビが引き続き受け皿レベルに留まっているか
