# 【検討過程の資料】enum対応表（MVP）

## 0️⃣ enumとは？

### １．本来の意味

**複数の定数をひとまとまりの型として定義するプログラミングの機能**

```markdown
from enum import Enum

class Energy(Enum):
LOW = 0
MEDIUM = 1
HIGH = 2
```

**何のためか？**

- 無効な値を入れにくくする
- コードの可読性を高める

---

### ２．広義のenum

**取りうる値があらかじめ決まっている選択肢**

```markdown
気力：低/中/高
料理結果：作った/簡単/温めた/外食...
```

**意味合い**

ユーザーの選択肢を意味のある集合体として扱う（＝「enum的」なデータとして扱う）

---

### ３．MySQLのenum型

```markdown
ENUMは、テーブル作成時に列挙された事前定義リストから選択された値を持つデータ型です。この事前定義リスト外の値を入力しようとするとエラーが発生します。
ENUMは文字列のように見えますが、内部的には整数として格納されています。
ENUMは文字列の可読性と整数のコンパクトなデータ型を兼ね備えています。

CREATE TABLE products (
  id INT AUTO_INCREMENT PRIMARY KEY,
  price ENUM('cheap', 'reasonable', 'expensive')
);

INSERT INTO products (price) VALUES ('cheap'), ('reasonable'), ('expensive');

SELECT price, price+0 FROM products;

+------------+--------+
| size       | size+0 |
+------------+--------+
| cheap      |      1 |
| reasonable |      2 |
| expensive  |      3 |
+------------+--------+
```

**enum型の利点**

- 無効なデータが入らない
- データベースの可読性向上（文字列で読める）

**enum型の難点**

- 値が変わるといちいちテーブルのスキーマを変更しなければならない
    - enum型は指定した各値を集合体としてカラムに定義するもの
    - 値を変更するのは例えばカラム名を変更するのと同じなので `ALTER TABLE` が必要になる
    - 上記の例で `cheap` `reasonable` `expensive` のほかに `middle` を追加するような場合
    - 逆にユーザーが `cheap` から `reasonable` へ選択を変更したい場合は `UPDATE`
- 移植性が低い
- 要するに後からデータの値が変わらないことを暗に要求している

---

### ４．今回のアプリへのあてはめ

TINYINT, BOOLEANはデータとして **enum的** であるので、 **広義のenum** として設計する。

- DBには文字列でなく数値で保存する
- Flaskで数値に意味を持たせる
- 対応表を作る

---

## 1️⃣ energy_level（気力）

**用途**

- 今日の状態
- 判断疲れの軽減・レコメンドの材料
- NULL = 選択しなかった

| DB値 | 定数名（例） | 表示文言 |
| --- | --- | --- |
| 0 | LOW | 低 |
| 1 | MEDIUM | 中 |
| 2 | HIGH | 高 |

**補足**

- 数値が大きいほど「余裕がある」
- 将来 `>=1` などの条件分岐がしやすい

## 2️⃣ result（今日の料理結果）

**用途**

- その日の「選択」を肯定するための中核
- NULL = 未入力（今日はまだ決めていない）

| DB値 | 定数名（例） | 表示文言 |
| --- | --- | --- |
| 0 | COOKED | 作った |
| 1 | NOT_COOKED | 作らなかった |
| 2 | BOUGHT | 買った |
| 3 | EAT_OUT | 外食 |
| 4 | FROZEN | 冷凍 |
| 5 | LEFTOVER | 作り置き |

**補足**

- 並び順に深い意味はない
- UIの表示順は自由に変えてよい
- `COOKED (0)` を基準に下層（粒度）を持たせる

## 3️⃣ cook_granularity（作った粒度）

**用途**

- result = COOKED のときのみ意味を持つ
- NULL = 未入力 / 該当なし

| DB値 | 定数名（例） | 表示文言 |
| --- | --- | --- |
| 0 | FULL | ちゃんと作った |
| 1 | SIMPLE | 簡単に済ませた |
| 2 | WARMED | 温めただけ |
| 3 | CUT_ONLY | 切っただけ |
| 4 | PLATED | 盛っただけ |

**補足**

- 「ちゃんと」が上、という **価値評価には使わない**
- あくまで **行為の粒度の違い**として扱う
- 表示文言は思想に合わせて後から調整可

## 4️⃣ time_minutes（所要時間）

※ enumではなく「実数値」だが、対応表を持つと実装が楽

| DB値 | 表示文言 |
| --- | --- |
| 10 | 10分 |
| 30 | 30分 |
| 60 | 60分 |

**補足**

- 数値のまま保存するのは良い判断
- 将来 20分・45分などを追加しやすい
