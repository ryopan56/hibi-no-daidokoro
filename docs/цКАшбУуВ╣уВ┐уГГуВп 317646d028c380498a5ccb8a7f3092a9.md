# 技術スタック

## 1. 結論

今回の最適解は、**Django 5.2 LTS + PostgreSQL + Django Templates + Bootstrap 5.3 + OpenAI Responses API** です。

理由は、このアプリが必要とするものが「認証」「フォーム」「画像アップロード」「セッション」「管理画面」「DBマイグレーション」「サーバーサイドで静かに成立するUI」であり、Djangoはそれらを最初から高い完成度で持っているからです。Django 5.2 はLTSで、Python 3.10〜3.14をサポートしています。

## 3. なぜ Flask ではなく Django か

Flaskでも作れます。

ただし今回は、ログイン認証、セッション、CSRF、フォーム、DBマイグレーション、画像アップロード、管理画面、通知設定、検索画面、エクスポート/インポートまで含みます。これをCodexで一気に実装するなら、**拡張を積み上げるFlaskより、最初から必要機能が揃っているDjangoの方が手戻りが少ない**です。Djangoの公式チュートリアル自体が、models、admin、forms、templates までを標準的なアプリ開発の流れとして扱っています。

私の判断では、**Flaskは“軽い”が、今回の要件には軽さが利点になりにくい**です。

一方Djangoは、日々の台所のような「ユーザーごとのデータ」「静かなUI」「堅い保存処理」が中心のアプリにかなり合っています。

## 4. バックエンド確定方針

### Django 5.2 LTS

これを採用します。

理由は3つです。

1. **LTSで安定している**
2. Python 3.12との相性が良い
3. Codexが生成するコードも、Djangoの“定石”に乗せやすい

Django 5.2 はLTSで、前のLTSである4.2のサポートは2026年4月に終了予定です。今から新規開発するなら、5.2を選ぶのが自然です。

### Python 3.12

これを採用します。

理由：

- 十分新しい
- Django 5.2が正式対応
- 周辺ライブラリの対応が安定している

---

## 5. データベース確定方針

### PostgreSQL 17

これを採用します。

理由：

- 本番運用に強い
- Djangoとの相性が非常に良い
- 制約・検索・拡張で困りにくい
- 将来の本番公開でも無難

PostgreSQLは主要バージョンごとに5年間サポートされます。

### なぜ MySQL ではなく PostgreSQL か

MySQLでも動きます。

ただ、今回はすでに **enumをDBに持たない** 方針なので、どちらでも大差はありません。

そのうえで私は、

- Djangoとの親和性
- 将来の検索・制約拡張
- 運用時の安心感

から PostgreSQL を推します。

### ローカル開発でもPostgreSQLを使う

ここは重要です。

今回は「後で本番向けにDBだけ差し替える」のではなく、**最初から本番寄りのDBで作る**方が良いです。

理由は、Codexに実装させる以上、環境差分が少ない方が事故が減るからです。

---

## 6. フロントエンド確定方針

### Django Templates + Bootstrap 5.3

これで行きます。

理由：

- このアプリはSPAにする必然が薄い
- 画面遷移が明確
- フォーム中心
- レンダリングの主役はサーバーでよい
- Codexが安定して作りやすい

Bootstrap 5.3 はフォーム部品とコンポーネントが充実していて、モバイルファーストのWeb UIを短時間で整えやすいです。

### React / Next.js を採用しない理由

今回は採用しません。

理由：

- 複雑すぎる
- 認証、フォーム、状態管理、API分離が増える
- 「静かな日記アプリ」に対してオーバースペック
- Codexで作れても、その後の保守コストが上がる

### htmx も今回は採用しない

htmx はサーバーHTML主体のまま部分更新ができる良い選択肢ですが、今回はまず **Django標準 + 最小JS** で十分です。

画面の本質が「入力→保存→表示」であり、複雑なリアルタイムUIが中心ではないためです。

## 7. AI連携確定方針

### OpenAI Responses API

これを採用します。

OpenAIの公式ドキュメントでは、**新規プロジェクトには Responses API を推奨**しています。

### モデル

**GPT-5 mini** を第一候補にします。

理由：

- 今回のAI出力は「長文の高難度推論」ではない
- 役割は短い提案文・肯定文の生成
- コストを抑えたい
- 回数制限前提でも、日次運用コストは小さい方がよい

OpenAIの公式モデル一覧では、GPT-5 mini は **faster, cost-efficient version of GPT-5 for well-defined tasks** とされています。

### Structured Outputs

使います。

理由：

- AI出力を毎回同じ構造にしたい
- Codex実装で壊れにくくしたい
- レコメンド3案や肯定メッセージのパースを安定させたい

OpenAIの Structured Outputs は、JSON Schemaに沿った出力を保証するための仕組みです。

### 料金の考え方

Responses API自体に別料金があるのではなく、**選んだモデルの入出力トークン料金で課金**されます。

### このアプリでAIを使う箇所

- 今日の最低限
- 高度レコメンド
- 初回保存時の肯定メッセージ
- 週次肯定メッセージ

### コスト対策

- 1日3回制限
- 同条件キャッシュ
- 失敗時は定型文フォールバック

## 9. 認証・セキュリティの確定方針

### 認証

- Django built-in auth を使用
- カスタムユーザーモデルを**最初から**作る

理由：

- 後からユーザーモデルを差し替えるのは面倒
- `name` や将来の設定項目を自然に持てる

### セキュリティ

- パスワードは Django 標準ハッシュ
- CSRF は Django 標準
- セッション認証
- 画像アップロードは拡張子とMIMEの両方を検証
- ファイル名はUUIDにする

## 10. 通知の確定方針

### “本物のPush通知”は今回やらない

代わりに、**アプリ内通知**にします。

理由：

- Web Push / メール通知は運用コストが上がる
- このアプリの思想に対して、押しつけ感が強い
- ローカル開発→限定公開→改善の流れに対して重い

### 実装

- 毎週水曜JST基準で判定
- 次回ホームアクセス時に表示
- ON/OFF設定あり

これは「通知体験」は残しつつ、実装負荷を下げる最適解です。
