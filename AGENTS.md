# AGENT.md

## 0. この文書の役割
この文書は、hibi-no-daidokoro リポジトリにおける Codex / ChatGPT の行動規範を定める。
この文書はプロダクト仕様書ではない。
プロダクト仕様・文書優先順位・横断仕様は SPEC.md を参照すること。
MCP や外部接続の実際の許可範囲は `.codex/config.toml` に従うこと。

## 1. 最優先ルール
- 仕様変更を勝手に行わない
- 仕様不足・仕様衝突を見つけた場合は、実装で吸収せず「提案」として整理する
- プロダクトに関する判断は SPEC.md を優先する
- ツール権限・外部接続は `.codex/config.toml` を優先する
- secrets は読まない、出さない、貼らない

## 2. 技術前提
- Django 5.x / PostgreSQL / Docker Compose / Django Templates
- `logs/app/` などの実行ログ系と、`logs/codex_sessions/` を運用する
- `logs/codex_sessions/` は Git 管理対象
- ローカル開発は Docker Compose 前提とする
- 既存のディレクトリ構成は勝手に変えない
- UI 関連ファイルは以下のディレクトリを使う
  - `docs/ui_pencil/`
  - `docs/ui_mockups/`
  - `docs/ui_specs/`

## 3. 作業開始時
作業開始時は、原則として以下を確認する。
- SPEC.md
- 現在の進捗サマリー
- 必要な仕様ページ
- Deep research レビュー
- `git status -sb`
- `git log --oneline --decorate -10`

必要な仕様ページとは、要件定義 / 非機能要件 / 画面仕様 / URLメソッド / テーブル定義 / enum対応表 / AIプロンプト仕様 / メッセージ仕様 などを指す。

フロントエンド作業では追加で以下を確認する。
- 承認済みモック画像
- 対応する実装メモ
- Pencil フロー採用時は対応する Pencil ファイル
- 画像 / 実装メモ / Pencil ファイルの version が一致していること

UI 関連ファイルの命名規則は以下とする。
- Pencil ファイル: `docs/ui_pencil/{screen-slug}_v{n}.pen`
- モック画像: `docs/ui_mockups/{screen-slug}_v{n}_approved.png`
- 実装メモ: `docs/ui_specs/{screen-slug}_v{n}.md`

最新版は以下のルールで判定する。
- 通常フローでは、対象画面について存在する `approved` 付きの最大 version を正とする
- Pencil フローでは、対象画面について存在する最大 version の
  - `.pen`
  - `_approved.png`
  - `.md`
  が揃っている組を正とする

Codex は古い version を使わないこと。

開始時の方針:
- まず前提を要約する
- 仕様衝突や前提不足があれば、実装前に列挙する
- 大きな UI 変更、本番インフラ構成、移行手順のような横断的作業は、いきなり実装せず先に計画を作る

## 4. 作業の基本姿勢
- 既存構成を尊重し、最小差分で変更する
- ディレクトリ構成や命名規則を勝手に変えない
- 不明点を推測で埋めない
- 破壊的変更は避ける
- 横断的な方針変更は提案として整理し、承認前に実装しない

## 5. Git 運用
- `main` は常に安定状態を保つ
- タスクごとに作業ブランチを切る
  - 例: `feat/task-12-frontend-planning`
  - 例: `fix/task-xx-...`
- push はユーザーがローカルで実施する
- Codex は `git push` を行わない
- PR 作成 / 更新のみ、ユーザーが明示的に依頼した場合に限り GitHub MCP 経由で行ってよい
- 破壊的な Git 操作は禁止
  - `git reset --hard`
  - `git clean -fd`
  - 履歴改変
  - 大量削除

## 6. 外部接続
- Codex 本体の一般ネットワークは原則 OFF
- 外部接続は `.codex/config.toml` の許可範囲に従う
- GitHub 操作は GitHub MCP に限定する
- Notion 連携は Notion MCP を用いてよいが、用途は原則として参照に限る
- Notion への更新・書き込みは、ユーザーが明示的に依頼した場合のみ検討する
- ただし、Notion 側 integration capability が read-only の場合は書き込みを試みない
- 許可されていない外部接続は行わない

## 7. Notion の扱い
- Notion は「作業ハブ / 要約ハブ / 補助参照元」として扱う
- 仕様の最終決定は SPEC.md に従う
- GitHub の最新状態を毎回そのまま Notion に転記しない
- Notion には要約を残す
- secrets や `.env` 実値は Notion に貼らない

## 8. secrets / 環境変数
- APIキー / PAT / DB接続情報 / `.env` の実値 / 各種 secret は表示しない
- 不要に `.env` を開かない
- `KEY` / `SECRET` / `TOKEN` を含む情報は原則として扱わない
- 例外として、PR 作成 / 更新を行う日だけ `GITHUB_PERSONAL_ACCESS_TOKEN` を一時的に使う運用は可とする
- secret 値を表示するコマンドは提案しない
- secret を含む内容を Notion / ChatGPT / ログへ転記しない

## 9. フロントエンド実装ルール
フロントエンド実装では、以下の 4 つを分けて扱う。
- 視覚設計の正本: Pencil ファイル（Pencil フロー時）
- 人間承認の正本: 承認済みモック画像
- 実装意図の正本: 実装メモ
- 実装の正本: コード

フロントエンド実装の基本フローは 2 つある。
- 通常フロー: ChatGPT → 承認 → Codex
- Pencil フロー: Pencil → 承認 → Codex

Pencil が使える場合は Pencil フローを優先候補とする。
Pencil が使えない場合、有料化した場合、PoC の結果が不十分な場合は通常フローに戻る。

フロントエンド実装時の最優先ルール:
- 承認済みモック画像がある場合は、その見た目を優先して実装する
- 実装メモがある場合は、その意図を優先して実装する
- Pencil フロー時は `.pen` を視覚設計の参照元として使う
- `.pen` と承認済み画像が矛盾する場合は、承認済み画像を優先し、その差異を明記する
- 画像で読み取れない部分は勝手に情報量や要素を増やさない
- 実装前に、画像 / 実装メモ / Pencil ファイルをもとにセクション分解・変更箇所整理を行う
- いきなりコーディングせず、まず構造化結果を提示する
- 構造化結果の確認前に、大きな UI 実装へ進まない

フロントエンド実装前に整理すること:
- 画面の主目的
- セクションの並び順
- 主 CTA
- 補助導線
- フラッシュメッセージやエラー表示位置
- モバイル表示時の縦積み方針
- 既存 URL / View / Form 契約を壊さないこと

フロントエンド実装時の禁止事項:
- 承認済みモックにない要素を勝手に増やすこと
- 1画面1目的を崩すこと
- AI 導線を主役にすること
- 見た目の大幅変更を、構造化確認なしで進めること

version ルール:
- v1 は最初に承認された版
- v2 以降は、承認済み見た目が変わったときだけ作る
- 文言修正、軽微な余白調整、バグ修正では version を上げない
- 通常フローでは、対象画面の最大 version の承認済みモックと実装メモを正とする
- Pencil フローでは、対象画面の最大 version の `.pen` / `_approved.png` / `.md` が揃っている組を正とする

## 10. テスト
- 変更後は適切なテストを行う
- 基本は `docker compose exec web python manage.py test` を用いる
- 必要に応じて対象を絞ったテストや手動確認を行う
- テスト未実施の場合は、未実施理由を明記する

フロントエンド実装後は追加で以下を行う。
- 実装画面のスクリーンショット比較を前提にする
- 承認済みモック画像と、実装後画面の差分を確認する
- Pencil フロー時は必要に応じて `.pen` とも見比べる
- 差分が大きい場合は v2 修正タスクとして扱う
- 実装直後の状態を最終デザイン確定とみなさない

## 11. 作業終了時
- 実装内容を確認する
- 必要なテストを実施する
- `git diff` で変更内容を確認する
- 必要に応じて `git diff --cached` でコミット対象差分を確認する
- 適切な粒度で `git add` / `git commit` を行う
- コミットメッセージは Conventional Commits を基本とする
- `logs/codex_sessions/` にセッションログを残す
- 完了報告には少なくとも以下を含める
  1. 変更ファイル
  2. 実装概要（仕様と実装の対応）
  3. 実行したテスト / コマンド
  4. 注意点 / 残課題
  5. ログファイル名

フロントエンド実装の完了報告では追加で以下を含める。
- 参照した承認済みモック画像名
- 参照した実装メモ名
- Pencil フロー時は参照した `.pen` 名
- 構造化結果を確認したこと
- 実装後のスクリーンショット比較結果
- 差分が残る場合は、その内容と理由

## 12. 禁止事項
- 仕様を勝手に変更すること
- secrets を読む / 出す / 貼ること
- Notion に secrets や生ログを貼ること
- GitHub へ勝手に push すること
- 破壊的な Git 操作をすること
- 未承認の外部接続を増やすこと
